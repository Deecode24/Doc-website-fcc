Introduction to Version Control

What is Version Control?

Version control is a system that records changes to a file or set of files over time so that you can recall specific versions later.

  • It allows multiple people to work on a project simultaneously, track changes, and revert to previous versions if necessary

    • Introduction to Version Control

    Definition:Version control is a system that records changes to a file or set of files over time so that you can recall specific versions later.

    Benefits:


  • Enables collaboration: Multiple people can work on a project simultaneously
  • Tracks changes: Keeps a history of modifications and updates.
  • Reverts to previous versions: Allows restoring previous versions in case of errors.

    • Types of Version Control Systems

    Centralized Version Control Systems (CVCS):

  • A single server stores all files and version history.
  • Clients check out files from this central server.
  • Examples: Subversion (SVN), Perforce.

    • Distributed Version Control Systems (DVCS):

  • Each client has a full copy of the repository, including its history.
  • Changes can be made and tracked locally.
  • Examples: Git, Mercurial.
    • Setting Up Git

    Introduction to Git

    Git is a distributed version control system created by Linus Tor

    Advantages of Git:

  • Speed and performance: Efficient handling of large projects.
  • Strong support for non-linear development: Branching and merging.
  • Data integrity: Ensures the integrity of source code with cryptographic hashing.
  • Distributed development: Each user has a complete copy of the repository.

    • Installing Git

    Windows:

  • Download Git from git-scm.com
  • Run the installer and follow the setup instructions.

    • macOS:

  • Install Xcode Command Line Tools by running the command below in the terminal
    • xcode-select --install
  • Alternatively, download Git from git-scm.com or use a package manager like Homebrew
    • brew install git

    Linux:

  • Use the package manager for your distribution (sudo apt-get install git for Debian/Ubuntu, for Fedora, etc.).
    • sudo dnf install git
    Basic Git Commands

    Initializing a Repository

    git init
  • Description: Creates a new Git repository in your current directory.

    • Cloning a Repository

    git add or git add

    Commiting changes

    git commit -m "Your commit message"
  • This records changes to the repository with a message describing the modifications.

    • Pushing Changes

    git push
  • Uploads local changes to a remote repository.

    • Pulling Changes

    git pull
  • Fetches and integrates changes from a remote repository to your local repository.

    • Configuring Git

    Setting up your identity:

    git config --global user.name "Your Name"
    git config --global --edit

    Setting up default text editor:

    git config --global core.editor "code --wait"

    Basic configuration commands:

    git config --list
    git config --global --edit
    Branching and Merging

    Understanding Branches

  • What is a branch?:A branch is an independent line of development in a repository.
  • Why use branches? Enables parallel development, feature isolation, and experimentation without affecting the main codebase.

    • Basic Branch Commands

  • Creating a branch:
    • git branch "branch-name"
  • Switching branches:
    • git checkout "branch-name" or git switch "branch-name"

    Merging branches:

    git merge "branch-name"
  • Integrates changes from the specified branch into the current branch.
    • Resolving Conflicts

    What are Merge Conflicts?

  • Merge conflicts occur when changes from different branches conflict and cannot be automatically resolved by Git.
  • Concurrent changes to the same lines of code or file modifications.

    • How to Resolve Conflicts

  • Identifying conflicts: Git will mark the conflicted areas in affected files.
  • Manual resolution: Edit the conflicted files to resolve differences.
  • Marking resolution: After resolving, mark the conflicts as resolved with git add and then commit the changes.
  • Using tools: Use merge tools like VS Code, P4Merge, or built-in Git tools to assist in resolving conflicts.
    • Using GitHub

    Introduction to GitHub

  • What is GitHub?: A web-based platform for version control and collaboration using Git

    • Benefits:

  • Hosting repositories: Store and manage code online.
  • Collaboration: Work with other developers on shared projects.
  • Project management: Track issues, manage tasks, and use project boards.

    • Working with Repositories

    Creating a repository:

  • Through GitHub interface: Click "New" on the repositories page, fill in details, and create.
  • From the command line: Create locally and then push to GitHub.

    • Forking a repository:

  • Fork button on a GitHub repository page.
  • It reates a copy of a repository in your GitHub account.

    • Pull requests

  • Creating a pull request: Propose changes to a repository by opening a pull request.
  • Reviewing and merging: Collaborators review the proposed changes before merging them.

    • GitHub Pages

    GitHub Pages is a service that hosts static websites directly from a GitHub repository.

    Setting up a GitHub Pages Site:

  • Create a repository named ".github.io."
  • Add your website files (HTML, CSS, JavaScript)
  • Push changes to the repository to publish the site.
    • Best Practices

    Writing Good Commit Messages

    Importance Clear and descriptive commit messages help in understanding the history and purpose of changes.

    Guidelines:

  • Use imperative mood: "Add feature" instead of "Added feature".
  • Keep messages concise but informative.
  • Include relevant details about what and why changes were made.

    • Commit Early, Commit Often

    Benefits

  • Smaller, more manageable changes.
  • Easier to track progress and isolate issues.
  • Better collaboration and code review process.

    • Using .gitignore

    PurposeSpecifies intentionally untracked files to ignore.

    Common Patternss

  • Ignoring build files and directories.
  • Ignoring sensitive information like API keys.
  • Ignoring OS-specific files and editor configurations.

    • Backup and Recovery

    Strategies for backing up repositories:

  • Regularly push local changes to a remote repository.
  • Use multiple remotes or backup services.

    • Recovering from accidental deletions or errors:

  • Use Git reflog to recover lost commits.
  • Restore deleted branches or files using Git history.
